Patrones de diseño


Los patrones de diseño son las mejores prácticas formalizadas que el programador puede usar para resolver problemas comunes al diseñar una aplicación o sistema. Los patrones de diseño pueden acelerar el proceso de desarrollo al proporcionar paradigmas de desarrollo probados y comprobados.

La reutilización de patrones de diseño ayuda a evitar problemas sutiles que pueden causar problemas importantes, y también mejora la legibilidad del código para codificadores y arquitectos que están familiarizados con los patrones.


Creational patterns

Builder

Nos permite construir objetos complejos paso a paso. El patrón nos permite producir distintos tipos y representaciones de un objeto empleando el mismo código de construcción.

implementación:

sacar el código de construcción del objeto de su propia clase y lo coloques dentro de objetos independientes llamados constructores.

El patrón organiza la construcción de objetos en una serie de pasos. Para crear un objeto, se ejecuta una serie de estos pasos en un objeto constructor. Lo importante es que no necesitas invocar todos los pasos. Puedes invocar sólo aquellos que sean necesarios para producir una configuración particular de un objeto.

Factory

Proporciona una interfaz para crear objetos en una superclase, mientras permite a las subclases alterar el tipo de objetos que se crearán.

implementación:

Se crea la interfaz que es común a todos los objetos, se crean los objetos que implementan dicha interfaz, la clase creadora declara el método fábrica que devuelve nuevos objetos, los Creadores Concretos sobrescriben el Factory Method base, de modo que devuelva un tipo diferente de producto.

Singleton

Nos permite asegurarnos de que una clase tenga una única instancia, a la vez que proporciona un punto de acceso global a dicha instancia.

implementación:

haciendo privado el constructor de la clase y creando un método que crea una instancia (objeto) de la misma si no existe. En Kotlin es aún más sencillo, haciendo uso del keyword object.



Structural patterns

Adapter

Permite la colaboración entre objetos con interfaces incompatibles. Se trata de un objeto especial que convierte la interfaz de un objeto, de forma que otro objeto pueda comprenderla.

Decorator

Permite añadir funcionalidades a objetos colocando estos objetos dentro de objetos encapsuladores especiales que contienen estas funcionalidades.

Fecade

Proporciona una interfaz que facilita el acceso a una biblioteca, un marco o cualquier conjunto complejo de clases. Una fachada es muy útil si tu aplicación solo necesita algunas de las funcionalidades de una biblioteca sofisticada entre las muchas que ofrece.



Behavioral patterns

Chain of responsibility

Permite que las solicitudes fluyan a través de una cadena de controladores. Cuando un controlador recibe una solicitud, decide si procesarla o enviarla al siguiente controlador de la cadena.

Iterator

Permite iterar a través de los elementos de una colección sin revelar su representación interna. El propósito del patrón de diseño del iterador es extraer el comportamiento que permite iterar a través de una colección y ponerlo en un objeto llamado iterador.

Strategy

La estrategia es un patrón de diseño de comportamiento que le permite definir una familia de algoritmos, colocarlos en clases separadas y hacer que sus objetos sean intercambiables. Permite tomar una clase que tiene un comportamiento específico pero lo realiza de diferentes maneras y divide sus algoritmos en clases separadas llamadas estrategias.
