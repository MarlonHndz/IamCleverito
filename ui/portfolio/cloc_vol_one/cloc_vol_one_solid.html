Principios SOLID


Los principios SOLID son eso: principios, es decir, buenas prácticas que pueden ayudar a escribir un mejor código: más limpio, mantenible y escalable.

En definitiva, prácticas para desarrollar un software de calidad.


> S : Single Responsibility Principle (SRP)


> O : Open/Closed Principle (OCP)


> L : Liskov Substitution Principle (LSP)


> I : Interface Segregation Principle (ISP)


> D : Dependency Inversion Principle (DIP)


En este sentido la aplicación de los principios SOLID está muy relacionada con la comprensión y el uso de patrones de diseño, que nos permitirán mantener una alta cohesión y, por tanto, un bajo acoplamiento de software.


S: Single Responsibility Principle (SRP)

“A class should have one, and only one, reason to change.”


Según el Principio de Responsabilidad Única “una clase debería tener una, y solo una, razón para cambiar”. Es esto, precisamente, “razón para cambiar”, lo que Robert C Martin identifica como “responsabilidad”.


Es decir, que cada clase o método sólo debe hacer una cosa, sencilla y concreta. Si un objeto tiene un solo cometido, éste será más fácil de mantener.


O: Open/Closed Principle (OCP)

“You should be able to extend a classes behavior, without modifying it.”


El Principio de Abierto/Cerrado dice “Deberías ser capaz de extender el comportamiento de una clase, sin modificarla”. En otras palabras: las clases que usas deberían estar abiertas para poder extenderse y cerradas para modificarse.


Si el día de mañana el cliente solicita un nuevo requisito para algo que ya tienes desarrollado, si tienes en cuenta este principio, el comportamiento de esa clase debería ser extendido, nunca modificado. En caso contrario la probabilidad de que te encuentres con daños colaterales es muy alta.


L: Liskov Substitution Principle (LSP)

“Derived classes must be substitutable for their base classes.”


El Principio de Sustitución de Liskov dice que “las clases derivadas deben poder sustituirse por sus clases base”.


Esto significa que los objetos deben poder ser reemplazados por instancias de sus subtipos sin alterar el correcto funcionamiento del sistema o lo que es lo mismo: si en un programa utilizamos cierta clase, deberíamos poder usar cualquiera de sus subclases sin interferir en la funcionalidad del programa.


I: Interface Segregation Principle (ISP)

“Make fine grained interfaces that are client specific.”


Según el Principio de Segregación de la Interfaz “Haz interfaces que sean específicas para un tipo de cliente”, es decir, para una finalidad concreta.


En este sentido, según ISP, es preferible contar con muchas interfaces que definan pocos métodos que tener una interface forzada a implementar muchos métodos a los que no dará uso.


D: Dependency Inversion Principle (DIP)

“Depend on abstractions, not on concretions.”


El último principio, el Principio de Inversión de Dependencias dice “Depende de abstracciones, no de clases concretas”.


El objetivo de este principio es conseguir desacoplar las clases de nuestro desarrollo. Que una clase pueda funcionar por sí sola sin depender de otra. Es decir que:

> Los módulos de alto nivel no deberían depender de módulos de bajo nivel. Ambos deberían depender de abstracciones.


> Las abstracciones no deberían depender de los detalles. Los detalles deberían depender de las abstracciones.
